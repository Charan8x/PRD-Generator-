Product Requirements Document (PRD)

01

Project Summary

Goal: Provide a unified dashboard and command line interface that analyzes code check-ins for styling issues, vulnerability exposures, and architectural anti-patterns, using lightweight local fine-tuned language model agents.

The system is tailored for teams seeking high security and low latency in their automated PR reviews, bypassing third-party cloud service vulnerabilities. This ensures zero source code leaks while supplying instant, actionable refactoring plans directly in developer terminals and central team dashboards.

02

Features

  • Local AI Static Analysis: Securely evaluate syntax, complexity, and performance defects within local offline parameters.
  • Interactive Git Hooks: Block commits automatically when critical security vulnerabilities or architectural violations are detected.
  • Central Team Dashboard: Aggregate metrics, code health history, and lint compliance across multiple code repositories.
  • Refactor Suggestion Interface: Apply suggestions with one click through our custom IDE panel extensions.
03

User Stories

  1. As a Developer, I want to receive immediate refactoring tips on git push so that I can address issues before sending a Pull Request.
  2. As a Security Engineer, I want to enforce static checks for secret keys and hardcoded passwords so that security flaws never reach production.
  3. As a Team Lead, I want to view trend charts of codebase health and styling compliance over time so that I can guide code quality reviews during sprints.
04

Database Design

The schema relies on standard relational tables supporting indexing on active user sessions and workspace tracking details.

-- Database Schema Outline (PostgreSQL)

CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE projects (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES users(id) ON DELETE CASCADE,
    name VARCHAR(100) NOT NULL,
    description TEXT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE prd_records (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id UUID REFERENCES projects(id) ON DELETE CASCADE,
    summary TEXT,
    features JSONB,
    database_design TEXT,
    api_specs JSONB,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

API Suggestions

We expose REST endpoints for authorization and schema creation. Below are the key endpoints:

// POST /api/v1/auth/signup
// Register a new user
Request: { "email": "dev@org.com", "password": "securepassword" }
Response: { "user_id": "8f89bc...", "token": "jwt-token-string" }

// POST /api/v1/prd/generate
// Start a new PRD generation task asynchronously
Request: { "project_name": "Antigravity Dev", "description": "Local AI static analysis..." }
Response: { "task_id": "t-8271a", "status": "processing" }

// GET /api/v1/prd/status/{task_id}
// Check processing state or return generated content
Response: { "task_id": "t-8271a", "status": "completed", "result": { ... } }
06

Test Cases

  • TC-101: Verify that registration fails with clear alerts when the email format is invalid or password length is under 8 characters.
  • TC-202: Confirm that the generate button transitions the page to a loading spinner state while disabled.
  • TC-303: Assert that error banners trigger when API calls return code 500, with a message advising the user to try again.
  • TC-404: Validate that the sidebar dynamically lists the 5 latest project history items, sorted by date in descending order.
07

Development Plan

The roadmap targets an initial product version in three sprints:

  1. Sprint 1: UI Shell & Auth integration (Week 1): Complete HTML/CSS screens, configure routes, and wire basic API login hooks.
  2. Sprint 2: Task Orchestration & AI Planner (Week 2): Connect model pipeline, configure database repositories, and add generation states.
  3. Sprint 3: Validation, History lists & Deploy (Week 3): Implement dashboard filters, write unit/integration tests, and build container builds.
